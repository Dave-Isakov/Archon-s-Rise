# M2.5 ŌĆö Win/Lose Systems + Doom-Driven Enemy Spawning ŌĆö Design

**Date:** 2026-07-07
**Milestone:** M2.5 (expanded: win/lose core + seeded enemy spawn zones + doom-scaled enemy difficulty)
**Build order:** one spec, two shippable phases ŌĆö Phase 1 win/lose core, Phase 2 spawning/scaling.

## Goal

Make a run winnable and losable, and make the Doom Clock *felt* on the map: as the clock
rises, enemies spawn from seeded zones at an escalating rate and get tougher. Closes the
deferred wound-hand loss rule. All balance numbers below are starting values ŌĆö tune in playtest.

## Architecture

Codebase idiom: **pure static rules classes + thin scene components** (matches `CombatRules`,
`PlaceRules`, `LevelRules`, `TurnButtonGate`). Pure classes are testable via the mcs CLI
harness; MonoBehaviours only wire scene state. All tuning knobs live on ScriptableObjects
(`LevelRewardsSO` pattern), inspector-editable with no code change.

New units:

| Unit | Kind | Purpose |
|---|---|---|
| `DoomRules` | pure static | band math: spawn cadence, tier gate, stat bonus, loss check |
| `RunEndRules` | pure static | the four end-of-run evaluations |
| `SpawnRules` | pure static | spawn-cell selection (candidates + rng ints in ŌåÆ cell out) |
| `DoomTuningSO` | ScriptableObject | every doom/spawn knob (max, bands, thresholds, bonuses) |
| `DoomClock` | MonoBehaviour | owns doom value; `Add(int)`; raises doom-changed; runs loss check |
| `EnemySpawner` | MonoBehaviour | round-end spawn execution from zones |
| `RunEndController` | MonoBehaviour | opens the run-end canvas, locks input, deletes save |

## Phase 1 ŌĆö Doom Clock + Win/Lose + Run-End Screen

### Doom Clock

- `DoomTuningSO` (single asset): `doomMax` (20), spawn-cadence bands, tier-gate thresholds,
  stat-bonus bands, and spawn-zone count (Phase 2 reads the same asset).
- `DoomClock` component holds the current doom value, exposes `Add(int amount)`:
  clamps to `doomMax`, raises a doom-changed event (HUD), then evaluates the doom loss.
- **Tick:** `GameManager.RoundPlus()` calls `DoomClock.Add(1)` ŌĆö the tick shares the exact
  cadence that refreshes units/skills; no event-ordering ambiguity.
- **API hook decision (2026-07-07):** nothing else calls `Add` in M2.5. Future event content
  (failed dungeon, ignored threat, +1..+3 pushes per balance.md) uses the same entry point.
- **HUD:** doom meter (fill bar + `12/20` text) on the main HUD canvas. Visual wiring done
  manually in-editor by Dave from step-by-step instructions (established workflow).

### Win/Lose evaluation ŌĆö `RunEndRules`

Four outcomes, each a pure function, each checked at its natural trigger point:

| Outcome | Rule | Checked when |
|---|---|---|
| **Win** | conquered castles Ōēź 2 | a conquest completes (`ConquestTracker`) |
| **Doom loss** | doom Ōēź `doomMax` | inside `DoomClock.Add` |
| **Wound-out loss** | total `StatType.Wound` cards across deck + hand + discard Ōēź 6 | immediately after wounds are added (combat loss, flee, assault retreat) |
| **Wound-hand loss** | hand tops up to full holding only wounds | after hand refill (turn/round end) |

- Wound adds are not undoable commands, so checking at add-time means the count can never be
  un-done back under threshold after the check fires.
- Wound-hand closes the deferred follow-up (memory note 2026-07-05): a hand of only wounds is
  unplayable ŌĆö now it's a loss instead of a stuck state.
- If a win and a loss land on the same event, **win is evaluated first**.

### Run-end screen

One canvas for all outcomes (`RunEndController`):

- Headline per outcome: *Victory!* / *The Land Has Fallen* (doom) / *Overcome by Wounds*
  (both wound losses).
- Run stats: rounds survived, castles conquered, player level.
- Single **Main Menu** button.
- Modal: input-locked like the validation message canvas; no game interaction underneath.
- **Deletes `Save.json`** on open ŌĆö a finished run cannot be resumed. (M3's run-setup flow
  will own new-run framing; M2.5 just returns to the main menu.)

## Phase 2 ŌĆö Seeded Spawn Zones + Doom Scaling

### Spawn zones

- `GridGeneration` seeds **4ŌĆō6 zone cells** across the whole 20├Ś20 map, deterministic from
  the map seed: not on/adjacent to a town cell, minimum distance from the start cell (0,0),
  minimum spacing from each other.
- **Replaces the accidental lower-left-only region** (`x 1ŌĆō10, y 2ŌĆō10`): initial map-gen
  enemies now also spawn from these zones, so one system governs all enemy placement from
  round 1. Initial spawns stay seed-derived; the defeated-cells save model still covers them.

### Spawn cadence (doom-banded)

At each round end, `DoomRules.ShouldSpawn(doom, roundsSinceSpawn)` applies the banded rate
from `DoomTuningSO` (starting values):

| Doom | Spawn rate |
|---|---|
| 0ŌĆō6 | every 3rd round |
| 7ŌĆō13 | every 2nd round |
| 14+ | every round |

A spawn picks a random zone, then a free cell in/adjacent to it via `SpawnRules` ŌĆö excluding
town cells, occupied enemy cells, and the player's cell. Saturated zones are skipped; if all
zones are saturated the spawn is skipped (no forced placement).

### Enemy scaling

- `EnemiesSO` gains `tier` (1ŌĆō3) and a stable `id`; a new Enemies `ContentRegistry` on
  `DataManager` (same pattern as Cards/Units/Skills). All 5 existing assets get ids; only
  the field enemies in `EnemyDeck.enemies` participate in zone spawning (assign them tiers
  1ŌĆō2). Dungeon enemies get ids and a tier but stay dungeon-only, outside the spawn pool.
- **Tier gating** (from `DoomTuningSO`): tier 1 always spawnable; tier 2 at doom Ōēź 7;
  tier 3 at doom Ōēź 14.
- **Flat stat bonus** at spawn (from `DoomTuningSO`): doom 7ŌĆō13 ŌåÆ +1 HP / +1 Attack;
  doom 14+ ŌåÆ +2 / +2.
- The bonus lives **on the `EnemyToken`** (`bonusHP` / `bonusAttack`), never written to the
  shared `EnemiesSO` asset. `EnemyCard` combat math and the enemy preview panel
  (`PreviewRules` consumers) both display **effective stats** (base + bonus).
- Map-gen (round-1) enemies spawn with zero bonus; only the doom value at spawn time matters.

### Save/load

- Alive mid-run spawns are captured in `spawnedEnemies` (cell, enemy content id, applied
  bonuses) at save time; load re-instantiates them directly through the Enemies registry.
- Chosen over deterministic replay (decision 2026-07-07): replay would break every in-flight
  save on any tuning change and requires identical RNG consumption forever.

## Testing

- CLI pure-harness (mcs ŌĆö see memory: MonoBleedingEdge `mcs.bat`, not legacy csc) tests for:
  - `DoomRules`: band boundaries (6/7, 13/14), cadence counting, clamp at max.
  - `RunEndRules`: each outcome's threshold edge; win-before-loss ordering.
  - `SpawnRules`: cell selection respects exclusions; saturated-zone skip.
- Scene verification: playtest checklist per phase (win a run, lose by clock, lose by
  wound-out, lose by wound-hand; spawns appear from zones, scale past doom 7/14, survive
  save/load).
- Canvas/HUD/prefab wiring performed manually in-editor by Dave from written instructions.

## Acceptance

- **Phase 1:** a run can be won by conquering 2 Castles and lost by clock-max, wound-out
  (Ōēź 6), or an all-wound hand; each shows the run-end screen and clears the save.
- **Phase 2:** enemies emerge from seeded zones on the doom-banded cadence, tougher tiers and
  stat bonuses appear past doom 7/14, and spawned enemies survive save/load.
