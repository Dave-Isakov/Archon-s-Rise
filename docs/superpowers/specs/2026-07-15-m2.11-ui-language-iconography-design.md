# UI Language & Iconography (M2.11) — Design (2026-07-15)

## Problem

The UI grew system-by-system, so each panel invented its own way to say the same things.
Costs render inconsistently: some are `icon + number`, some bare numbers, some prose. It is
not obvious at a town that recruiting **spends Influence**; the enemy pay-off, crystal
purchase, empower, and dungeon delve costs each dress differently. There is no canonical
icon per core concept — the HUD, cards, and panels don't always agree — so a new player
cannot transfer what one screen taught them to the next screen.

This milestone exists to make the game *legible to a stranger*: one icon per concept, one
layout dialect for costs and actions, everywhere. It is also a hard prerequisite for
**M2.12 (Tutorial & Help)**: tutorial and help text will embed these icons inline (TMP
sprite tags), which only teaches if the same glyphs appear in the real UI.

## Goals

1. **`IconRegistry` — one source of truth.** A ScriptableObject mapping every core concept
   to its sprite and its TMP sprite-asset tag name. Code and prefabs reference the
   registry; rich text writes `<sprite name="...">` and renders the same glyph the HUD uses.
2. **Complete concept list.** The 8 stats (Attack, Defend, Explore, Influence, Heal, Wound,
   Crystal, Siege), the 4 crystal colors + wild, Doom, XP/Level, HP (toughness), army/unit
   slot, place types (Town / Keep / Castle), Dungeon. Missing sprites are authored by the
   user (art is a user task; the registry just points at sprites).
3. **The layout language** — four rules, applied everywhere:
   - **Costs are always `[icon][number]`** — recruit prices, crystal purchase, empower,
     dungeon delve costs, enemy influence pay-off. Never bare numbers, never prose-only.
   - **Action buttons are always `[icon] Label`** — Assault, Recruit, Heal, Delve, Buy.
   - **The four action stats always list in the same order** (Attack, Defend, Explore,
     Influence) wherever they appear together.
   - **One "can't afford / locked" treatment** — the dim+lock style already used on unit
     options becomes the global standard for every unaffordable/disabled entry.
4. **The audit sweep.** Apply the four layout rules panel-by-panel: TownMenu, RecruitPanel,
   DisbandPanel, DungeonPanel, UnitPickerPanel, EnemyPreviewPanel, CardMenuCanvas
   (empower / improvise sections), RewardCanvas, skill bar, HUD, run-end screen.
5. **Editor-workflow discipline.** All prefab/scene wiring is delivered as step-by-step
   Unity-editor instructions for the user to perform (standing practice — no hand-edited
   scene/prefab YAML).

## Non-Goals (explicitly deferred)

- **No visual redesign** — no new theme, fonts, palette, or layout rework beyond the four
  rules. This is a consistency pass, not an art pass.
- **No controller/focus work** (separate roadmap thread; targets stay mouse-first for now).
- **No tutorial content** — banners, help entries, and the ? icon are M2.12.
- **No localization scaffolding** — strings stay inline; the registry only canonicalizes
  icons, not copy.

## The Model

### IconRegistry

`IconRegistrySO` (one asset) holds an entry per concept:

- `concept` — an `IconConcept` enum value (one member per Goal-2 concept).
- `sprite` — the Sprite used by `Image`-based UI.
- `tmpSpriteName` — the name the glyph resolves to inside the TMP Sprite Asset
  (`<sprite name="influence">`).

Access is via a small static wrapper (`IconRegistry.Get(IconConcept)`) that loads/caches
the asset, so any script or prefab-bound component can resolve a concept without scene
references. All registry sprites must live in (or be added to) the **default TMP Sprite
Asset** so `<sprite name=...>` tags resolve in every TMP text without per-text setup —
this extends the TMP sprite-asset setup already in use.

### Layout language details

- A reusable **`CostRow` prefab** (icon `Image` + TMP number) is the standard cost widget;
  panels adopt it during the sweep rather than styling their own.
- Unaffordable state on a `CostRow` or list entry uses the standard dim+lock treatment
  (same component/styling as unit-option locks) so "I can't do this yet, and here's the
  price" reads identically at towns, dungeons, pickers, and the card inspector.
- Stat groups (enemy preview, HUD, reward summaries) render in the fixed order
  Attack → Defend → Explore → Influence; Siege/Heal/Crystal/Wound appear after, when present.

### Audit checklist (per panel)

For each panel in Goal 4, the sweep asks three questions and fixes only what fails:
1. Does every cost render as a `CostRow` (`[icon][number]`)?
2. Does every action button carry its concept icon?
3. Do stat lists follow the canonical order, and locked entries the standard treatment?

The sweep is deliberately **surgical** — panels that already conform are left untouched.

## Error handling

- **Missing registry entry** — `IconRegistry.Get` logs an error and returns a visible
  placeholder sprite (never null/invisible), so a miss is loud in play testing.
- **Unresolvable TMP tag** — caught by validation tests (below), not at runtime.

## Testing

- **EditMode validation tests** (pure data checks, own asmdef per house pattern):
  - every `IconConcept` enum member has a registry entry with a non-null sprite and a
    non-empty `tmpSpriteName`;
  - every `tmpSpriteName` resolves to a glyph in the default TMP Sprite Asset (no dead
    `<sprite>` tags waiting to happen).
- **Manual visual pass** — the audit checklist above, walked panel-by-panel in the editor
  after wiring; acceptance is the checklist passing on every listed panel.

## Acceptance

- `IconRegistrySO` exists, is complete for the Goal-2 concept list, and validation tests
  are green.
- HUD and every audited panel reference registry icons — no bare-number cost anywhere in
  the Goal-4 panel list.
- `CostRow` is the shared cost widget on every panel that shows a price.
- A TMP text containing `<sprite name="...">` for any registry concept renders the same
  glyph the HUD shows for that concept.
- M2.12 can quote any concept inline in banner/help text by tag name alone.
