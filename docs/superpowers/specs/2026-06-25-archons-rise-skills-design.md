# Archon's Rise — Design & Roadmap Skills (Spec)

_Date: 2026-06-25_
_Status: approved design, pending implementation plan_

## Goal

Produce two committed, project-level Claude skills that (1) capture the game's design as a
living game-design document (GDD), and (2) maintain a living development plan for finishing the
game. The skills travel with the repo so any future session — or contributor — loads the same
context when designing content or deciding what to build next.

This spec covers building the **skills**, not implementing the game systems they describe. Game
systems are tracked as milestones inside the roadmap skill and specced/planned separately when
their turn comes.

## Validated game vision (decisions log seed)

These decisions were made during brainstorming on 2026-06-25 and are the canonical seed for the
GDD and the decisions log:

1. **Structure: roguelike runs.** Short, self-contained runs against escalating difficulty on a
   randomized hex map. Replayability over a single long campaign.
2. **Win condition: Rise to Archon (domination).** A run is won by accumulating power rather than
   beating a single boss — a concrete threshold combining territory and growth (e.g. control N
   towns AND reach a Level/Influence target). Exact numbers are tuning, defined in `balance.md`.
3. **Lose condition: wounds + doom clock.** Two pressures:
   - **Wounds (tactical):** losing combat shuffles Wound cards into the deck (deck pollution);
     too many (deck clog / HP threshold) ends the run. Uses existing Wound/Mend cards.
   - **Doom clock (strategic):** a rising threat advances each round; if it maxes before the
     player is crowned Archon, the run is lost. New system.
4. **Meta-progression: content unlocks only.** Runs start from identical conditions. Between runs
   the player permanently unlocks new cards/units/enemies/events that enter the future-run pool.
   No power carryover.

## Existing systems (from code review, 2026-06-25)

Already implemented in code and authored as ScriptableObjects:
- Four action stats (Attack/Defend/Explore/Influence) + Heal/Wound/Crystal, generated by playing
  cards each turn; turn/round structure; full undo (Command pattern).
- Cards with Empower via colored Crystals (Red/Yellow/Green/Purple); 7 starting + ~9 acquirable.
- Hex-map exploration (6 directions, Explore cost) via `GridGeneration` (randomized).
- Combat (player Attack vs enemy HP; enemy Attack vs Defend → wounds); field + dungeon enemies.
- Towns (Town/Village/Fortress/City) and recruitable Units (Knight/Scout/Warrior/Merchant).
- Dungeons; Rewards (Experience/Crystals/Cards at Beginner→Master tiers); leveling
  (code hints: even level = +1 stat, odd = +HP, every 3 = +hand size, every level = new skill).
- ScriptableObject GameEvent/Listener bus; JSON save (currently scalar-only — see roadmap M1).

Missing / stubbed (the work the roadmap orders): run-based save/load, win condition (Archon
threshold), doom clock, wound-out loss, run setup (seed), meta-unlock pool, plus the
Important-tier code-quality items from the code review.

## Deliverable: two project-level skills

Location: `.claude/skills/` inside the repo (committed).

### Skill 1 — `archons-rise-design` (the GDD / reference)

Loaded when designing content or mechanics. Stable; changes slowly.

| File | Contents |
|------|----------|
| `SKILL.md` | One-paragraph pitch, index of the other files, and the **design pillars** — the constraints every design decision is checked against. Frontmatter `description` triggers on game-design / content-authoring work. |
| `mechanics.md` | The locked decisions: run loop, win = Rise to Archon (threshold), lose = wounds + doom clock, turn/round flow, four stats, empower/crystal economy, leveling curve. |
| `content-rules.md` | The **authoring contract** per ScriptableObject type (valid Card/Enemy/Town/Unit/Reward/Dungeon), grounded in the actual SO fields, so new content stays consistent and is correctly generatable. |
| `balance.md` | Number ranges, reward/rarity tiers, crystal costs, doom-clock pacing, Archon threshold values, the unlock pool. |

**Pitch (canonical):** _A single-player roguelike deckbuilder where you explore a randomized hex
realm, build a crystal-empowered deck, and race a doom clock to Rise to Archon before wounds or
the falling land stop you._

### Skill 2 — `archons-rise-roadmap` (the living plan)

Loaded when deciding what to build next. Changes often.

| File | Contents |
|------|----------|
| `SKILL.md` | How to use the roadmap + a **current focus** pointer. Frontmatter `description` triggers on planning / "what's next" / status work. |
| `status.md` | What exists vs. stubbed vs. missing, seeded from the code review. |
| `milestones.md` | Ordered milestones to a playable roguelike loop (see below). |
| `decisions-log.md` | Append-only design decisions with rationale, seeded with the four decisions above, so reasoning survives across sessions. |

### Milestone ordering (approved)

1. **M1 — Run-based save/load.** Finish deferred code-review Critical #3, scoped to "persist a run
   in progress" (deck/hand/discard as card ids, map seed, doom-clock + run state) rather than a
   sprawling world. Make save explicit + on quit. Prerequisite for everything.
2. **M2 — Win/lose systems.** Archon threshold (win), doom clock (strategic loss), wound-out
   (tactical loss). Turns the existing sandbox into a winnable/losable run.
3. **M3 — Run setup & meta-unlocks.** Run seed/initialization, the content-unlock pool, and the
   between-runs unlock flow.
4. **Later milestones (fold in after M1–M3):** content expansion (cards/enemies/towns/dungeons per
   the authoring contract) and the Important-tier refactors from the code review
   (`docs/code-review.md`): event-driven updates over per-frame `Update()`, de-coupling
   gameplay→UI via events, the apply/revert toggle refactor, asmdefs + EditMode tests,
   modernization pass.

## "Living" maintenance contract

For the skills to stay useful rather than rot:
- The **roadmap** `SKILL.md` keeps a single **current focus** line; `milestones.md` is reordered
  as priorities shift; `decisions-log.md` is append-only.
- When a design decision is made or changed, it is recorded in `decisions-log.md` and reflected in
  the relevant GDD file in the same change.
- When a milestone completes, `status.md` and `milestones.md` are updated to reflect reality.
- These update rules are written into each `SKILL.md` so any session knows to maintain them.

## Out of scope

- Implementing any game system (save schema code, doom clock, win check, etc.) — those are
  milestones, specced/planned individually later.
- The Important/Minor code-review items — captured as later roadmap milestones, not done here.
- Exact balance numbers — placeholders with sensible ranges; tuned during implementation.

## Success criteria

- Both skills exist under `.claude/skills/`, are committed, and load via their `description`
  triggers when doing design vs. planning work.
- The GDD encodes the four validated decisions and the authoring contract for every existing SO type.
- The roadmap encodes status, the M1→M3 ordering, and the seeded decisions log, with written
  maintenance rules.
- A fresh session, loading only these skills, can correctly (a) author a new valid card/enemy/town
  and (b) state what to build next and why.
