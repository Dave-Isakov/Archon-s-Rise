# Tutorial & Help System (M2.12) — Design (2026-07-15)

## Problem

The game is being handed to external playtesters who will play unassisted. Today nothing
teaches the core loop (cards → stats → act), and several systems are invisible until
stumbled into (recruiting spends Influence; wounds pollute the deck; doom is a clock).
A confused first minute costs a playtester; a confused first hour costs the playtest.

The design constraint is the game itself: maps are randomized and systems surface
organically (skills at level 2, dungeons when found, doom bands mid-run), so a scripted
"tutorial level" fights the game's structure. The tutorial must therefore be **contextual
and event-driven**, teach on the player's *real* first run, and stay out of the way of
anyone who already knows the game.

Depends on **M2.11 (UI Language & Iconography)**: all tutorial/help text embeds registry
icons inline (`<sprite name="...">`), which only teaches because the same glyphs appear in
the real UI.

## Goals

1. **Guided first round ("the rail").** On a fresh profile, an ordered step sequence walks
   round 1: play a card → pools fill → move → fight a nearby weak enemy → end turn — each
   step advancing off the **real game event**, never a faked or locked interaction.
2. **Reactive one-shots.** Systems met after round 1 get one short banner each, fired once
   per profile by their real trigger (first Wound, first crystal, first level-up, first
   town, first dungeon, deck-can't-refill, doom entering the flagged band).
3. **On-demand help everywhere.** Every major panel carries a **?** icon opening a short
   explainer — what this screen is, what you can do, what it costs — with registry icons
   inline. The ? pulses the first time its panel type opens.
4. **Fully optional.** A Skip button on the rail banner; a "Tutorial tips" settings toggle;
   a "Reset tutorial" action. Tutoring off = one canvas off; the game is never modified.
5. **Starter-enemy spawn guarantee.** Map generation always places at least one tier-1
   enemy within early reach of the start ring — the rail's fight step must always have a
   subject (and it's a good pacing rule for every run).
6. **No save-schema bump.** All tutorial state is device-level (PlayerPrefs), not run state.

## Non-Goals (explicitly deferred)

- **No input locking / forced clicks.** Steps wait; they never block other actions.
- **No authored tutorial map or fixed seed.** The rail runs on the real first run.
- **No playtest shell** (build packaging, feedback links, version stamp) — separate
  follow-up milestone.
- **No controller flow** — mouse-first, but `TutorialTarget` ids are input-agnostic so a
  controller-focus pass can reuse them later.
- **No localization** — copy lives on the SO assets, inline.

## The Model

### TutorialCanvas

One new **Screen Space – Camera** canvas (matching the existing canvases), sorted above
all gameplay canvases. It hosts three prefabs and nothing else:

- **Step banner** — instruction text (TMP, registry icons inline) + **Skip Tutorial**
  button. Top-center by default, never covering the hand or the highlighted target.
- **Highlight frame** — a pulsing outline rect that parks itself over the current step's
  target (found by id via `TutorialTarget`). No dim-with-hole overlay — cheap and robust
  across canvas scaling.
- **Help popup** — the shared reader every ? icon opens (title + body, close on X or
  outside click).

Tutoring disabled = this canvas inactive. Nothing in existing canvases moves.

### TutorialManager and the rail

`TutorialManager` (a scene manager like the others, always active — **it** hosts all event
listeners, never the banner/popup children, avoiding the self-disabling-listener trap)
runs an ordered list of **`TutorialStepSO`** assets:

- `id` — stable string (persistence key component).
- `bannerText` — TMP rich text with registry sprite tags.
- `highlightTargetId` — optional; matches a `TutorialTarget` in the scene.
- `completion` — either **Next-button** (informational steps) or a **GameEvent reference**
  (the step completes when that event fires).

**Out-of-order tolerance:** when the rail starts, the manager begins counting every
completion event it references. When a step becomes current, if its event has already
fired since rail start, the step auto-completes immediately. The rail can therefore never
stall because the player explored ahead of the script.

**Draft rail (content, tunable as SO assets):**
1. *Welcome* — "Conquer 2 Castles before the Doom Clock fills." Highlights doom meter. (Next)
2. *Play a card* — highlights hand → completes on card-played.
3. *Your pools* — highlights the four stat icons; one line on improvise. (Next)
4. *Move* — highlights reachable hexes → completes on player-moved.
5. *The enemy* — highlights the guaranteed starter enemy; mentions hover preview →
   completes when combat starts.
6. *Fight* — "Attack ≥ HP defeats it; weak Defend means Wounds." → completes on that
   enemy's resolution (fight, siege, or influence pay-off all count).
7. *End turn* — "stats reset, hand refills" → completes on turn-end.
8. *Send-off* — "The rest is yours to discover — tap any ? when a screen is new." (Next)

Rail completion (or Skip) marks the profile tutored; the rail never runs again unless
reset.

### Reactive one-shots

`TutorialOneShotSO`: `id`, `bannerText`, optional `highlightTargetId`, `trigger`
(GameEvent). Fired at most once per profile, only while tutoring is enabled, and only
after the rail is done (rail steps outrank one-shots; a one-shot whose trigger fires
mid-rail is deferred and shown after the send-off, oldest first — even if slightly stale,
each points at the panel's ? for the durable explanation). Each one-shot stays short. Launch set: first Wound,
first crystal (empower explainer), first level-up, first town entered, first dungeon
entered, deck-can't-refill (must end round), doom entering the flagged band.

### TutorialTarget

A dumb component dropped on any UI element: `string targetId`. Registers itself in a
static registry on enable, unregisters on disable. The highlight frame resolves ids
through the registry, so the tutorial never hard-references other canvases' internals,
and prefab wiring stays trivial (add component, type id).

### Help entries and the ? icon

- **`HelpEntrySO`** — `panelId`, `title`, `body` (TMP rich text with registry icons).
- **`HelpIcon` prefab** — a small ? button dropped into a panel's corner, holding its
  `HelpEntrySO` reference; opens the shared help popup. Pulses until its entry has been
  opened once (per-profile seen set), then rests.
- Launch entries: place menu (services, conquest, buy crystals), recruit (per-unit
  Influence prices, army cap, disband-to-hire), disband, dungeon (3 delves, explore cost,
  exp-only fights, completion bundle, flags), unit pop-out (options, exhaust, costs),
  enemy preview (Normal vs Siege vs Influence resolutions), card inspector (empower,
  improvise, convert), rewards, skill bar (cadence, exhaust), HUD (stats, doom, deck/hand),
  run-end screen.

**Modal interplay:** banners and the help popup are *not* modals and never enter the
`RewardQueue` (same precedent as `UnitPickerPanel`). To avoid visual fights, the banner
and any open help popup **hide while a RewardQueue modal or picker canvas is open** and
reappear after. The help popup is player-initiated, read-only, one at a time.

### Persistence & settings

PlayerPrefs only (device-level, run saves untouched):
- `tut.enabled` (default 1), `tut.railStep` (index; −1 = done/skipped),
- `tut.oneshot.<id>` seen flags, `tut.help.<panelId>` seen flags.

Control surface: there is no settings screen yet, so both controls live on the existing
in-game **MainMenu** (`Assets/Scripts/Menu/MainMenu.cs`, which already hosts Save): a
**Tutorial tips** toggle (master switch for rail + one-shots + pulses; ? icons themselves
always remain) and a **Reset tutorial** button (clears all keys — for demos and shared
machines). A future settings screen can absorb them unchanged. Skip button = rail and
one-shots off (marks done), ? icons stay. Quitting mid-rail resumes at the saved step
index next session. This is the codebase's first PlayerPrefs use — keys are namespaced
`tut.*` to stay collision-free.

### Starter-enemy spawn guarantee

A pure rule in map generation: after the normal enemy pass, if no tier-1 enemy lies within
the configured starter radius of the start ring, place (or retier) one at a valid cell in
that radius — never on places, dungeons, or the start ring itself, respecting existing
spawn-blocking rules. Radius and the on/off knob live with the other spawn tuning.

## Error handling

- **Highlight target missing** (id not registered — panel closed, prefab renamed): the
  banner still shows, the frame hides, a warning logs. A missing target never blocks
  progression.
- **Completion event never fires** (player wanders off): the rail waits indefinitely —
  soft guidance, no timeouts, no forced actions.
- **Tutoring toggled off mid-step:** banner and frame clear immediately; state keys keep
  their values so toggling back on resumes cleanly.
- **PlayerPrefs wiped:** the rail simply runs again — harmless by design.
- **Reward modal opens mid-banner:** banner hides, reappears when the queue drains (see
  Modal interplay).

## Testing

- **Pure `TutorialRules` class** (own asmdef folder + EditMode tests, house TDD pattern):
  step-advance logic, already-fired auto-completion, one-shot dedupe + defer-past-rail,
  skip semantics, seen-set behavior — all as pure state transitions.
- **Spawn guarantee**: pure-rule EditMode tests (no starter enemy → one placed in radius;
  one already present → generation unchanged; blocked cells respected).
- **Registry-dependent copy**: a validation test that every sprite tag used in
  `TutorialStepSO` / `TutorialOneShotSO` / `HelpEntrySO` text resolves in the TMP sprite
  asset (extends the M2.11 validation).
- **Manual playthrough checklist**: fresh profile → full rail; skip path; toggle path;
  quit/resume mid-rail; each one-shot; each ? entry; banner-vs-reward-modal interplay.

## Acceptance

- Fresh profile: the rail teaches round 1 hands-off, tolerates out-of-order play, and
  never stalls; Skip works instantly and permanently.
- Every listed panel has a working ? with icon-inline copy; the ? pulses only until first
  read.
- One-shots each fire exactly once, never during the rail, never when disabled.
- Tutorial toggle and reset behave as specced; quit/resume mid-rail resumes at the same
  step; run saves are byte-identical in schema (v6, unchanged).
- 20/20 generated maps contain a tier-1 enemy within the starter radius.
- All EditMode suites (TutorialRules, spawn guarantee, copy validation) are green.
